Multi-Tier Web Application Deployment
Production-style 3-tier WordPress deployment on AWS — reverse proxy, web (Apache+PHP), private DB (MySQL).

Architecture

Internet → Reverse Proxy (Nginx) → Web Server (Apache+PHP) → Database Server (MySQL)

Public Subnet: 10.0.1.0/24 — reverse-proxy (public).
Private Subnets: web 10.0.2.0/24, db 10.0.3.0/24 — internal communication only.

VPCEC2NAT GatewaySecurity Groups

Project Overview

Goal: Deploy a production-style 3-tier WordPress application using AWS EC2 focusing on reliability, security, and scalability. Manual AWS console steps were used to create VPC, subnets, IGW, NAT, route tables and security groups, then EC2 instances were launched and configured.

Setup & Runbook

Step 1
VPC & Subnets
Create VPC (10.0.0.0/16), public-subnet (10.0.1.0/24), web (10.0.2.0/24), db (10.0.3.0/24).
Step 2
IGW & Route Tables
Attach IGW to public RT; private uses NAT for outbound updates.
Step 3
EC2 Instances
Launch reverse-proxy (public), web-server (private), database-server (private).
Step 4
App Stack
Install MySQL on DB, Apache+PHP on web, Nginx on reverse-proxy + proxy_pass.
Step 5
Security
Configure SGs: proxy-sg, web-sg, db-sg with minimum access.

Troubleshooting & Fixes

Major issues and fixes encountered during deployment:

  • SSH access failed — relaunch instance with Auto-assign Public IP or use bastion + ProxyJump.
  • Key pair mismatch — verify .pem name & permissions (chmod 400).
  • DB connection refused — ensure MySQL bind-address and SGs allow internal traffic.
  • Missing files / default nginx page — sync WordPress files to /var/www/html/wordpress and set ownership to www-data.

Techniques Learned

  • Manual VPC, subnet and RT setup to emulate production segmentation.
  • Using ProxyJump for SSH into private instances.
  • Database security with SG references and host-specific MySQL users.
  • Automated backups using cron + mysqldump.

Achievements & Next Steps

Achievements
